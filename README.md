# olhaeolhae - AI 협업 실험 프로젝트

## 프로젝트 개요

이 프로젝트는 **AI와 어디까지 협력할 수 있을지 알아보는 실험적인 프로젝트**입니다. 간단한 생일/연락처 관리 서비스를 만들면서 사람과 AI가 어떤 방식으로 협업할 수 있는지, 그 한계와 가능성을 탐구하고 있습니다.

---

## AI 협업 개발 환경의 진화

### 진화 타임라인

이 프로젝트는 Claude Desktop + JetBrains MCP에서 시작하여 Claude Code로 전환하며 AI 협업 방식을 지속적으로 발전시켜왔습니다.

```
 2025년 초         2025년 6~10월         2025년 12월          2026년 1월~
     │                 │                    │                   │
     ▼                 ▼                    ▼                   ▼
[by-ai] 태그       문서 기반 협업          Skills 시스템      이슈 기반 + Actions
     │                 │                    │                   │
     ▼                 ▼                    ▼                   ▼
"AI가 짠 코드"     "지침 문서 읽고 따라"    "필요한 지침만 로드"   "작업 추적 + 자동 리뷰"
```

#### 단계 1: 초기 협업 (`[by-ai]` / `[with-ai]` 태그)
- **시기**: 2025년 초
- **방식**: Claude Desktop + JetBrains MCP의 조합: 클로드 코드와 유사한 방식으로 코드 생성
- **규칙**: AI가 전부 담당하면 `[by-ai]`, 사람과 의견을 주고받으며 구현하면 `[with-ai]`를 커밋에 자동으로 붙이도록 지시
- **발견한 현실**:
  - 결과물의 품질을 위해선 작업 지시 전 컨텍스트 전달, 관리 등 **모든 과정에 사람 개입 필수**
  - `[by-ai]`작업물은 사람이 파악 못한 **블랙박스**로서 검수 오버헤드, 프로젝트 통제 불안감만 증폭
  - "해줘" 식 지시 → **상세한 작업 정의 + 컨텍스트 전달**로 작업 방식 변화
- **결론**:
  - 결국 모든 작업이 `[with-ai]`일 수밖에 없음 → **태그를 떼기로 결정**
  - **"누가 코드를 작성했느냐"**보다 **"코드가 설계한 구조 안에서 올바르게 구현되었는가"**가 더 중요한 질문

#### 단계 2: 문서 기반 협업
- **시기**: 2025년 6월 ~ 10월
- **방식**: 상세한 개발 지침 문서 작성, Claude가 문서를 읽고 따르도록 유도
- **특징**:
  - `be/instructions/` 폴더에 6개 이상의 지침 문서 (2,000줄+)
  - PM-001 ~ PM-025 태스크로 워크플로우 표준화
  - FT-001 ~ FT-018 태스크로 기능 구현
- **한계**: 지침 파일 간 충돌, CLAUDE.md 62.9KB로 비대해짐

#### 단계 3: Skills 시스템 도입
- **시기**: 2025년 12월 27일
- **도입 이유**:
  - CLAUDE.md가 비대해지면서 **세션 시작 시 지침만으로 메모리 과다 점유**
  - 오히려 잘 동작하지 않는 비효율 발생
  - **Compact conversation을 늦추고, 세션 컨텍스트를 효율적으로 사용**하기 위한 시도
- **방식**:
  - CLAUDE.md 경량화 (62.9KB → 2KB, 96.8% 감소)
  - **필요한 시점에 필요한 지침만 로드** (스킬 기반)
  - CLAUDE.md에는 **언제나 지켜야 할 지침만** 간단히
- **특징**:
  - 8개 스킬로 작업 맥락별 지침 분리
  - `/core-tdd`, `/infra-tdd` 등 명령어로 필요한 지침만 로드
  - 7가지 상충 규칙 통일

#### 현재: 두 가지 시도
- **시기**: 2026년 1월~

**1. PRD를 GitHub 이슈로 대체**
- **배경**:
  - 스킬이 원하는 타이밍에 알아서 로드되길 바랐지만, 잘 동작하지 않을 때가 많음
  - AI가 개발을 주도하는 프로젝트에서 **진행 현황을 놓치고 길을 잃는 느낌**
- **목적**:
  - 스킬 사용 범위 명료화
  - 작업 단위를 이슈로 분리하여 진행 상황 추적

**2. Claude Code GitHub Actions 도입**
- **배경**: CodeRabbit 등 여러 LLM 서비스를 다 잘 써야 한다는 심적/비용적 부담
- **결정**: **Claude Code 하나로 다 활용**
- **구성**:
  - PR 생성 시 자동 코드 리뷰, `@claude` 멘션 응답
  - 5가지 전문가 관점 리뷰 (백엔드, 프론트엔드, UX, DevOps, 비용 최적화) - **내가 모르는 키워드를 최대한 많이 학습하기 위해**

## 인간 역할의 전환: 코드 작성자에서 설계자/검증자로

### 체감하고 있는 변화

이 프로젝트를 진행하면서 인간의 역할이 명확하게 전환되고 있음을 몸소 체험하고 있습니다:

| Before | After |
|--------|-------|
| 직접 코드 작성 | 아키텍처/정책 설계 |
| 문법 오류 수정 | 설계 의도 검증 |
| 구현 디테일 결정 | 비즈니스 규칙 정의 |
| "어떻게 구현하지?" | "이게 내가 원한 건가?" |

**인간의 새로운 역할**:
- **디자인**: UI/UX 방향성 결정, 사용자 경험 설계
- **기획**: 기능 요구사항 정의, 우선순위 결정
- **정책 설계**: 명명 규칙, 아키텍처 패턴, 코딩 표준 수립
- **검증**: AI가 구현한 코드가 설계한 구조 안에서 올바르게 동작하는지 확인

### AI와 "말이 통하는 상태" 만들기

협업 과정에서 중요한 깨달음을 얻었습니다:

> **AI와 효과적으로 소통하려면, 인간이 먼저 "넓은 범위를 본 적"이 있어야 한다.**

`/wrap-lesson` 스킬은 이 깨달음에서 탄생했습니다:

```
/wrap-lesson
```

이 스킬은 세션에서 작업한 내용을 **"아기에게 설명하듯"** 깊이있게 분석하여 학습 문서를 생성합니다.

**왜 이런 스킬이 필요한가?**

1. **용어 기반 소통**: 구구절절 설명하기보다 "페이드 그라데이션", "WCAG 2.1"처럼 **용어 하나로 명확하게** 지시해야 AI와 커뮤니케이션 오류가 없습니다.

2. **넓고 얕은 학습**: 모든 분야를 깊이 알 수는 없지만, **최소한 키워드는 알아야** AI가 구현한 것을 보고:
   - 뭘 했는지 파악할 수 있고
   - 어디가 이상한지 알아챌 수 있고
   - 무엇을 질문해야 할지 알 수 있고
   - 간결하고 명료하게 요청할 수 있습니다

3. **"대화가 된다"**: AI가 구현한 디자인, FE, BE, Infra... 이 모든 것들을 보고 **검수 포인트를 알아챌 수 있는 상태**가 되어야 진정한 협업이 가능합니다.

### 올해의 실험

저는 올해 다음을 테스트하고 있습니다:

1. **넓고 얕은 학습**: 깊이보다 넓이. 한 분야의 전문가가 되기보다 **AI와 대화할 수 있는 범위를 최대한 넓히는 것**.

2. **내가 읽기 좋은 코드 구조 만들기**: 역할이 설계자/검수자로 바뀌었으니, **역할을 더 잘 수행할 수 있는 환경**을 만드는 것이 중요.

3. **다양한 코드 구조 많이 읽기**: 다양한 구조에 익숙해져서 **검수 속도를 높이는 것**도 필요.

## 현재 AI 협업 방법론

### TDD 기반 스킬 워크플로우

```
/write-prd → /prd-to-test → /core-tdd → /infra-tdd → /presentation-tdd → /commit → /create-pr
     │            │              │           │              │              │           │
     ▼            ▼              ▼           ▼              ▼              ▼           ▼
  PRD 작성    테스트 도출    코어 계층    인프라 계층   프레젠테이션     커밋 작성    PR 생성
 (전문가검토)               TDD 구현    TDD 구현      TDD 구현
```

### 사용 가능한 스킬 (8개)

| 스킬 | 용도 |
|------|------|
| `/write-prd` | 시니어 전문가 관점에서 PRD 검토 및 작성 |
| `/prd-to-test` | PRD를 테스트 케이스로 변환 |
| `/core-tdd` | Domain, UseCase, Service 계층 TDD |
| `/infra-tdd` | JpaEntity, Adapter, Repository TDD (실제 DB 사용) |
| `/presentation-tdd` | Controller, DTO Outside-In TDD |
| `/refactor` | 테스트 통과 상태 유지하며 리팩토링 |
| `/commit` | 커밋 메시지 작성 |
| `/create-pr` | AI 리뷰 최적화 PR 생성 |
| `/wrap-lesson` | 세션 학습 회고 및 문서화 |
| `/tasks` | 업무 분석 및 우선순위 제안 |
